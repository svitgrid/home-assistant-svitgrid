## 1. Background & motivation

Today the Svitgrid Home Assistant add-on is a **relay**: it reads existing HA
sensor entities (produced by some other HACS integration) and forwards their
values to the cloud. SP-A built a neutral, data-driven **register-spec**: one
JSON document per inverter model, generated from the canonical Dart
`inverter_protocol` maps, that says how to read each field from Modbus registers.
SP-A is merged (monorepo local `main`) and serves specs at
`GET /api/v1/register-specs/:modelId`, but nothing consumes them yet.

**SP-B makes the add-on read the inverter directly.** It adds a Python engine
that connects to the inverter over Solarman-V5 (port 8899) or raw Modbus TCP
(Victron/Huawei, port 502), reads the registers named by the spec, decodes them
to the canonical reading fields, and feeds the **same** reading payload into the
**same** SQLite store â†’ drain â†’ cloud pipeline the relay path already uses. The
user no longer needs a separate third-party inverter integration in HA.

SP-B is **reads only**. Control writes are SP-C; the polished phoneâ†’add-on
onboarding handoff is SP-D.

### Initiative context (not all in scope here)

## 3. Architecture & components

New package `custom_components/svitgrid/harvest/`, each module one
responsibility, all unit-testable offline.

### 3.1 `register_spec.py` â€” spec model

Parses the register-spec JSON (the shape SP-A's `RegisterSpecSchema` validates):
`modelId, version, source, verified, protocol, port, defaultSlaveId, flags,
reads[], derivations[], writes[]`. Mirrors the Dart `RegisterSpec`. `writes` are
parsed but ignored in SP-B (SP-C consumes them). A light `validate()` rejects an
unknown `op`/`builtin` or a dangling derivation input (defence in depth; the API
already validated on store).

### 3.2 `decoder.py` â€” the heart

`decode(spec, raw) -> dict[str, float | None]` where `raw` is
`{unitId: {address: rawWord}}` (mirrors the Dart `RawRegisters`).

A faithful port of the SP-A Dart **reference decoder**:
- 16-bit read decode: sentinel (`0xFFFF`â†’0 unsigned, `0x7FFF`â†’0 signed), sign
  extension at â‰Ą32768, then `value*scale+offset`.
- 32-bit (`words:2`): combine `(hi<<16)|lo` from `addr`/`addr+1`, 32-bit sign
  wrap at `0x80000000`, then scale/offset.
- Closed derivation ops: `product, sum, negate, passthrough, scale, builtin`.
- Named built-ins: `pv_power_from_vi, battery_sign_normalize, battery_temp_clamp,
  phase_voltage_grid_or_load, phase_load_ct_or_inverter, grid_relay_bit,
  daily_grid_unavailable` â€” each implemented once, keyed by name.

**Plus the reader sanity-clamps** that SP-A deliberately left out of the data
format (spec Â§3.5): `pvPower >= 0`, `|batteryPower| > 50000 â†’ 0`,
`batteryTemperature` kept only within `[-20, 80]`, etc. These make the Python
output match the **real reader** (the golden-vector ground truth), not the
clamp-free reference decoder. They live in a small `sanitize()` step applied
after `decode`, so the clamp set is explicit and testable in isolation.

`null` means absent (sentinel hit, missing register, clamp-rejected) â€” the
payload builder omits absent fields, so nothing fabricated ships.

### 3.3 `transport.py` â€” wire clients

`read(spec, conn) -> raw` selected by `spec.protocol`:
- **Solarman V5** (`pysolarmanv5.PySolarmanV5`): constructed with the inverter
  IP, `loggerSerial`, port 8899; `read_holding_registers(addr, count)` per range.
  Solarman data loggers commonly close the socket after each response, so the
  engine opens a fresh connection per poll (mirrors the Dart reader's
  per-request model).
- **Modbus TCP** (`pymodbus` sync client): `read_holding_registers(addr, count,
  slave=unitId)` (and `read_input_registers` for FC04). Victron addresses carry
  per-unitId (`100/247/245`); Huawei uses one slave + FC03.

**Range batching:** the engine groups the spec's read addresses into contiguous
ranges (respecting the Modbus max-registers-per-read cap and per-unitId
grouping), reads each range once, and assembles the `raw` map. Batching is
derived from the spec data â€” no per-model code.

All `pysolarmanv5`/`pymodbus` calls are synchronous socket I/O and MUST run via
`hass.async_add_executor_job` so the HA event loop never blocks (the same rule
the SQLite store already follows).

### 3.4 `engine.py` â€” one-poll orchestration

`async def poll_once(hass, spec, harvest_config, store, inverter_id)`:
1. read raw via transport (in executor),
2. `fields = sanitize(decode(spec, raw))`,
3. `payload = build_reading_payload(inverter_id, source_fields=fields)`,
4. `await store.append(payload)`.

A `run_direct_harvest_loop` background task (spawned per inverter that has
`harvest_config`) calls `poll_once` on the server-driven cadence, mirroring the
existing `run_readings_loop` structure (`while not hass.is_stopping and
lifecycle.active`, `asyncio.sleep(next_sleep_s)`, fail-soft on error).

### 3.5 Payload reuse (refactor)

`readings_publisher.build_reading_payload()` currently reads HA entity states.
Refactor it so the field-collection step is one function and the
payload-assembly (timestamp, `inverterId`, `source`, the `pv1Power â†’ pvPower1`
API renames, `pvPower` aggregation) is another that takes a `{field: value}`
dict. The HA-entity path and the harvest engine both call the assembly step. One
payload contract, two sources, converging at the same store. (`source` stays
`"edge"` â€” the cloud treats HA and direct-harvest readings identically.)

## 4. Cross-repo contract: golden vectors

The correctness backbone. The Python decoder must reproduce the Dart reader's
decoded output for every spec.

### 4.1 Generation (monorepo)

A new Dart tool `packages/inverter_protocol/tool/export_golden_vectors.dart`
runs each SP-A contract fixture through the **real reader's `decodeFromRaw`** (the
proven ground truth â€” SP-A already pins reference-decoder == reader) and emits
`packages/inverter_protocol/golden-vectors.json`: a list of
`{ modelId, spec, rawRegisters, expectedFields }`, where `expectedFields` is the
reader's decoded output (including sanity-clamps). Committed and guarded by a
staleness gate (regenerate-and-diff, like SP-A's spec staleness test).

The vector set MUST include, beyond the per-model daylight fixtures, at least:
- one **clamp-exercising** case (e.g. a battery power > 50000 or temp out of
  `[-20,80]`) so the sanitize step is locked,
- one **sentinel** case (`0xFFFF`/`0x7FFF`),
- one **32-bit** case (Huawei) covering hi/lo combine + sign.

### 4.2 Distribution & vendoring (this repo)

The committed `golden-vectors.json` is **vendored** into this repo at
`tests/fixtures/golden-vectors.json`. A header field records the source commit +
a content checksum; a tiny test asserts the vendored file's checksum matches its
header (so an out-of-date copy is loud, not silent). Re-vendoring is a documented
manual step (`scripts/sync-golden-vectors.sh` copying from the sibling monorepo
checkout). Not API-served â€” these are test fixtures, not runtime data.

### 4.3 Python contract test

`tests/harvest/test_golden_vectors.py` loads the vendored vectors and asserts
`sanitize(decode(spec, raw)) == expectedFields` (within float tolerance) for
every vector. This pins the Python engine to the Dart reader's actual output â€”
derivations, builtins, sentinels, 32-bit, and clamps â€” without the Python test
needing Dart.

## 5. Config & spec fetch

### 5.1 `harvest_config`

Per inverter in `entry.data["inverters"][i]`:
```python
"harvest_config": {
    "protocol": "solarman_v5",   # | "modbus_tcp"
    "ip": "192.168.1.100",
    "port": 8899,                 # 8899 Solarman, 502 Modbus TCP
    "slaveId": 1,
    "modelId": "deye_sg04lp3",    # selects the register-spec
    "loggerSerial": "1234567890", # Solarman only; "0" for Modbus TCP
}
```
SP-B adds a minimal config-flow step to enter these fields (protocol, IP, port,
slave, model picker, logger serial). SP-D will populate them from the phone
handoff instead and can drop or hide this manual step.

### 5.2 Spec fetch/cache

`api_client.get_register_spec(modelId)` mirrors `get_preset()`:
`GET /api/v1/register-specs/:modelId` â†’ the full spec JSON. Cached by `version`
in `entry.data` (like `merged_preset_version`). Fail-open: on fetch failure keep
the last good spec; if none is cached yet, skip the poll and log â€” never crash
the loop. Re-checks version periodically (reusing the preset-refresh cadence).

## 6. Error handling

Mirror the existing readings loop's fail-soft posture:
- A poll failure (TCP timeout, malformed frame, missing spec, decode error) is
  caught, logged, backs off to the default interval, and retries on the next
  cycle. It never tears down the task.
- An unreachable inverter simply produces no readings (the store isn't appended)
  â€” the same end state as a stale HA entity today.
- Sentinel/invalid registers and clamp-rejected values decode to absent; absent
  fields are omitted from the payload (nothing fabricated).
- The harvest loop is isolated per inverter; one inverter's failure doesn't
  affect another's loop or the relay path.

## 8. Open questions / risks

- **`pysolarmanv5`/`pymodbus` fidelity vs. the Dart transport.** The libs return
  raw register words; our decoder owns all scaling/derivation, so semantics live
  in the (golden-vector-pinned) decoder, not the lib. Risk is confined to
  framing/addressing (which range to read, FC03 vs FC04, unitId) â€” covered by
  transport tests + a live smoke on `deye_sg04lp3` later.
- **Vendored-vector staleness.** The checksum-header test makes a stale copy
  loud, but the cross-repo sync is manual. Acceptable for a test fixture; if it
  becomes a burden, revisit API-serving the vectors.
- **Sanity-clamp completeness.** The clamp set must match the readers exactly;
  the golden vectors (with clamp-exercising cases) are the oracle. Any clamp the
  reader applies that the vectors don't exercise could drift â€” planning will
  enumerate the reader clamps and add a vector for each.
- **Unverified addresses.** SP-B proves Python==Dart; it does not prove the
  addresses are right. Every non-`deye_sg04lp3` model stays `verified:false`
  pending a hardware register_dump (same posture as SP-A).
